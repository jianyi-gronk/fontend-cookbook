## 1. 前端工程化

- 前端工程化是一种实践，它通过使用工具、框架和流程来简化前端开发过程，提高生产效率并确保代码质量。随着 Web 技术的快速发展和前端项目变得越来越复杂，前端工程化变得非常重要。前端工程化主要涉及以下几个方面：
  - **模块化（ 将代码按照功能或逻辑拆分成多个独立的、可重用的模块的过程）和组件化（ 将用户界面（UI）拆分成多个独立的、可重用的组件的过程 ）**：将代码拆分为可重用的模块和组件，有助于提高代码复用性、降低维护成本并提高开发效率。前端工程化通过引入模块化和组件化的概念，使开发人员能够更好地管理和维护大型项目。
    - 相应工具：Webpack、Rollup、Parcel 等
  - **构建和打包**：构建过程包括将源代码转换为可在浏览器中运行的代码，例如将 TypeScript 转换为 JavaScript、将 Sass 转换为 CSS、代码压缩和优化等。打包是将多个模块和资源组合成一个或多个输出文件的过程，以便在浏览器中加载和运行
    - 相应工具：Webpack、Rollup、Parcel、Gulp、Grunt 等
  - **自动化测试**：自动化测试可以帮助开发人员快速发现并修复代码中的错误，从而提高代码质量。前端工程化通常包括单元测试、集成测试和端到端（E2E）测试等多种类型的自动化测试
    - 相应工具：Jest、Mocha、Karma、Cypress、Protractor 等
  - **代码质量检查**：为了确保代码质量和一致性，前端工程化通常包括使用代码质量检查工具，如代码风格检查、静态类型检查和代码复杂度分析等
    - 相应工具：ESLint、Prettier、TSLint、TypeScript 等
  - **持续集成和持续部署（ CI/CD ）**：持续集成是将代码更改频繁地合并到主分支的过程，通常伴随着自动化测试和代码质量检查。持续部署是将代码更改自动部署到生产环境的过程。通过实现 CI/CD，开发团队可以更快地交付新功能并修复错误，同时确保代码质量
    - 相应工具：Jenkins、Travis CI、CircleCI、GitHub Actions 等
  - **开发工具和环境**：前端工程化还包括使用一系列开发工具和环境来提高开发效率，例如使用代码编辑器、调试工具、版本控制系统和代码生成器等
    - 相应工具：Visual Studio Code、Chrome DevTools、Git、Angular CLI、Create React App、Vue CLI 等

#### 前端基建

- 前端基建是前端工程化中的一个组成部分，它主要关注于为前端开发团队提供一套标准化、可复用的基础设施和工具。前端基建通常包括项目脚手架、开发环境配置、构建和部署流程、代码规范和格式化工具、组件库和样式库等。前端基建的目标是简化开发过程，使开发者能够专注于业务逻辑，而不是花费大量时间在环境搭建、配置和优化上。

## 2. 管理依赖项工具

#### npm

- npm 是 Node.js 的默认包管理器，已经成为了 JavaScript 生态系统的标准工具
- npm 的依赖关系管理方式可能导致项目的 node_modules 目录变得庞大，因为它会为每个依赖项的版本创建一个单独的副本。这可能会导致更长的安装时间和更大的磁盘空间占用
- npm 在安装依赖项时会自动生成一个 package-lock.json 文件，用于确保项目的依赖项版本一致
- 适用于大多数 JavaScript 项目，因为它是最广泛使用和支持的包管理器

#### npm 工作区（ Workspaces ）

- 是 npm v7 及更高版本引入的一项功能，用于管理具有多个子项目（子包）的单个代码库。它允许您将项目结构组织为一个主项目和多个子项目，这些子项目可以共享相同的依赖项。这对于在同一个代码库中管理多个相关的包（例如，具有前端和后端组件的项目或一个包含多个插件的库）非常有用，使用 npm 工作区有以下优点：
  - 依赖项管理：所有子项目可以共享相同的 node_modules 文件夹，这有助于减小项目的总体大小，提高依赖项安装的速度
  - 便捷的脚本管理：您可以在主项目中定义脚本，以便在所有子项目中运行
  - 简化的版本控制：使用单个代码库可以更轻松地跟踪和管理多个相关项目的更改
  - 设置 npm 工作区：
    - 在主项目的 package.json 文件中添加一个 workspaces 属性，该属性是一个包含子项目路径的数组（或者通过 workspace.yaml）
    - 在每个子项目中创建一个 package.json 文件，以便它们被识别为独立的 npm 包
    - 运行 npm install，它将在主项目中创建一个共享的 node_modules 文件夹，其中包含所有子项目的依赖项
    - 可以在主项目中使用 npm run 命令运行子项目的脚本，或者使用 npm publish 发布子项目。请注意，要使这些命令适用于特定子项目，您需要在子项目的 package.json 文件中定义相应的脚本和配置

#### pnpm 和 yarn

- pnpm 和 yarn 提供了更好的性能和对大型项目的支持
- pnpm
  - pnpm 是 npm 的一个替代品，专注于提高性能和减少磁盘空间占用
  - pnpm 通过使用一个全局存储和使用硬链接/符号链接的方式来共享依赖项，避免了重复安装相同的依赖项。这可以显著减少磁盘空间占用和安装时间
  - pnpm 使用 pnpm-lock.yaml 文件来锁定依赖项版本，类似于 npm 的 package-lock.json 文件
  - pnpm 支持工作区，允许在一个单一的仓库中管理多个项目，并共享依赖关系。这在 monorepo 结构中非常有用
  - 由于 pnpm 相对较新，某些工具和库可能不完全兼容，但它已经在许多项目中得到了成功应用
  - pnpm 使用 workspace.yaml（或叫 pnpm-workspace.yaml） 文件进行工作区配置
- yarn
  - Yarn 是由 Facebook 开发的一个 npm 替代品，专注于提高性能和安全性
  - Yarn 使用一个离线缓存和并行安装来加速依赖项的安装过程
  - Yarn 使用 yarn.lock 文件来锁定依赖项版本，类似于 npm 和 pnpm 的锁文件
  - Yarn 支持工作区（类似于 pnpm），允许在一个单一的仓库中管理多个项目，并共享依赖关系。这在 monorepo 结构中非常有用
  - Yarn 在 JavaScript 生态系统中得到了广泛的支持和应用，尤其是在 React 和 React Native 社区中
  - Yarn 使用 package.json 文件中的 workspaces 字段进行工作区配置

#### npx

- npx 是 npm 的一个附属工具，用于在不全局安装的情况下执行 Node.js 包（即命令行工具，比如 Vue CLI，ESLint，Prettier 等），它可以直接运行项目中已安装的依赖包的可执行文件，而无需指定完整的路径或将其安装为全局工具
- npx 还允许直接运行远程代码库（例如 GitHub 仓库）中的代码，而无需先安装到本地。当需要执行一次性任务或尝试新工具时，npx 非常方便，相比于 npm 安装到本地的区别
  - 临时性：当您使用 npx 从 GitHub 下载并运行代码时，它会在一个临时目录中下载代码，然后立即执行。执行完成后，临时目录将被清除。这意味着代码不会长期留在您的本地计算机上。相反，当您将代码安装到本地时，它会被永久保存在您的项目或全局依赖中，直到您手动删除它
  - 隔离性：由于 npx 只在临时目录中下载并运行代码，因此它不会干扰您项目的其他依赖。这在尝试新工具或一次性任务时非常有用，因为您无需担心它们对项目产生潜在影响。而在本地安装代码时，您可能需要考虑版本冲突、重复依赖等问题
  - 便捷性：npx 使您可以快速尝试和运行远程代码，而无需经过安装、配置和卸载的过程。这在评估新工具或执行一次性任务时非常方便。与之相比，安装到本地需要更多的步骤和时间
